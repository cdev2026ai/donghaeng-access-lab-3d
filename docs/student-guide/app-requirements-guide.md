# 앱 요구사항 도출 가이드

## 체험 기록을 앱 기능으로 바꾸는 방법

시뮬레이터에서 느낀 어려움은 그대로 앱 기능이 될 수 있습니다.

```text
불편했던 상황
→ 부족했던 정보
→ 앱이 판단해야 할 조건
→ 사용자에게 전달할 안내
```

## 예시 1. 신호 시간이 부족함

| 항목 | 내용 |
|---|---|
| 상황 | 고령 보행자가 신호가 시작된 뒤 출발했지만 시간 내 횡단이 어려움 |
| 부족한 정보 | 현재 속도로 횡단 가능한지 알기 어려움 |
| 앱 기능 | 남은 신호 시간과 예상 횡단 시간 비교 |
| 안내 예시 | “현재 속도로는 시간이 부족합니다. 다음 신호를 기다리거나 보행 지원이 필요합니다.” |

## 예시 2. 경사로 위치를 찾기 어려움

| 항목 | 내용 |
|---|---|
| 상황 | 휠체어 사용자가 단차 앞에서 막힘 |
| 부족한 정보 | 경사로 위치와 거리 |
| 앱 기능 | 경사로 방향·거리 안내 |
| 안내 예시 | “오른쪽 2m에 경사로가 있습니다.” |

## 예시 3. 점자블록이 끊김

| 항목 | 내용 |
|---|---|
| 상황 | 시각장애인 사용자가 점자블록 단절 구간에서 방향 확신도가 낮아짐 |
| 부족한 정보 | 다음 유도선 위치, 현재 방향 |
| 앱 기능 | 방향 유지 안내와 다음 유도선 위치 안내 |
| 안내 예시 | “정면에서 왼쪽 10도 방향으로 2m 이동하면 점자 유도선이 다시 이어집니다.” |

## 예시 4. 불법주차 차량으로 통로가 좁아짐

| 항목 | 내용 |
|---|---|
| 상황 | 차량이 보도를 막아 통로 폭이 좁아짐 |
| 부족한 정보 | 통과 가능 여부, 우회 가능성 |
| 앱 기능 | 장애물 폭, 남은 통로 폭, 우회 경로 판단 |
| 안내 예시 | “전방 차량으로 통로가 좁아졌습니다. 우측 통로 폭을 확인하고 천천히 이동하세요.” |

## 요구사항 문장 템플릿

```text
[사용자 유형]은/는 [상황]에서 [부족한 정보] 때문에 이동 판단이 어렵다.
따라서 앱은 [판단 기준]을 바탕으로 [안내 정보]를 [안내 방식]으로 제공해야 한다.
```

예시:

```text
고령 보행자는 횡단보도에서 남은 신호 시간과 자신의 보행 속도를 비교하기 어렵다.
따라서 앱은 남은 신호 시간과 예상 횡단 시간을 비교해 출발 가능 여부를 음성·자막으로 안내해야 한다.
```
